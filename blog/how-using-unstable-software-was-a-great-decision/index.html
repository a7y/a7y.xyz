How using unstable software was a great decision

I use Laravel. It is an extremely popular PHP framework, and it has the definite right to be. Laravel is simply a pleasure to work with.

About 4 months ago I started planning a new project. I had to make the important decision of whether or not to use the unstable, not-even-beta Laravel 5, or the tried and true Laravel 4. I ended up choosing 5 for a couple of reasons.

  • Its release date (4 months ago) was set for December. I was sure I wouldn’t have finished development by then anyway.
  • Even the 5 or 6 new features shown back then were extremely attractive. Method injection, request classes, and the new directory structure were some of the few. n.b. present-day Laravel 5 has a much larger list of stellar improvements.
  • I am a sucker for bleeding edge software. Just consult my loyalty to Arch Linux.

I realized that the absolute worst that could happen was an increased requirement of time. A serious liability for teams and corporations, but thankfully, a non-issue for me. I’m working alone, and hence, set my own pace and deadlines.

Going for Laravel 5 was seriously a fantastic decision. I cannot express the amount of progress I’ve made in regard to my code and the way I conceptualize projects. I would go so far as to say that every pitfall associated with alpha-software resulted in either the improvement or learning of a skill.

Documentation was virtually non-existent. If I didn’t know how to use a feature, the only option I had was to read through Taylor Otwell’s code. This went to great lengths in improving my understanding of a number of design patterns, and of course, the Laravel internals.

A changelog was also absent. A nice-looking markdown document with functionality changes and improvements? Nope. Everyone got the commit list and file diffs. If something broke that worked previous to a > composer update, I went to GitHub and looked at what Taylor pushed.

Constant additions, reversions, and revisions kept me on my feet, or rather, in my chair. Especially with the changes to convention, it was a rare week when I hadn’t refactored anything. I can definitely understand why people would loathe this, but I’d be lying through my teeth if I said it didn’t result in the betterment of my code.

Bugs in the framework? I either fixed them myself or waited for someone else to do so. Doing the former, I can truthfully say that a project with 14k+ stars on GitHub incorporates code I wrote. Although insignificant when compared to others’ contributions, it’s a great feeling regardless.

So overall, I firmly believe I made the right choice. I maintain the notion that unstable software is horrible for production. However, it now carries the exception that it’s incredible for personal-growth.

Now if you’ll excuse me, I need to go make my code PSR-2 compliant.
